Create `pool-app/src/components/photo-upload.tsx` as a "use client" component.

**Props:** `{ jobId: string; onPhotoAdded?: () => void }` — onPhotoAdded allows parent to refresh gallery after upload.

**Two hidden file inputs (per locked decisions):**
1. Camera input: `<input type="file" accept="image/jpeg,image/png" capture="environment">` — per PHOT-01, opens iPad camera. Do NOT use `accept="image/*"` (Safari 17+ would deliver HEIC). Do NOT add `multiple` (camera captures one at a time).
2. Library input: `<input type="file" accept="image/jpeg,image/png,image/webp" multiple>` — per PHOT-02, allows selecting several photos at once from library.

**Two visible buttons (per locked decisions):**
1. "Take Photo" button with Camera icon — triggers camera input click. Use `min-h-[56px] text-lg gap-2` for iPad touch target.
2. "From Library" button with ImagePlus icon, variant="outline" — triggers library input click. Same sizing.
Buttons arranged in a `flex gap-3` row.

**Processing pipeline (per locked decisions and research Pitfall 4):**
Process files SEQUENTIALLY (not Promise.all) to avoid iPad memory pressure. For each file:
1. Check isHeicFile() — if true, dynamically import heic2any: `const heic2any = (await import("heic2any")).default;` and convert to JPEG blob, then wrap in a new File object with .jpg extension and image/jpeg type.
2. Compress with `imageCompression(file, COMPRESSION_OPTIONS)` from browser-image-compression. Import COMPRESSION_OPTIONS from @/lib/photos.
3. Upload compressed file with `upload()` from `@vercel/blob/client`:
   - `access: "public"`
   - `handleUploadUrl: "/api/photos/upload"`
   - `onUploadProgress: ({ percentage }) => updateProgress(filename, percentage)`
4. After upload resolves, call `savePhotoMetadata(jobId, { url: result.url, filename: originalFilename, size: compressed.size })`.
5. Call `onPhotoAdded?.()` to signal the parent to refresh.

**Upload state tracking:**
Use `useState<UploadStatus[]>` where UploadStatus is:
```typescript
type UploadStatus = {
  id: string;        // unique ID (crypto.randomUUID or Date.now + filename)
  filename: string;
  progress: number;  // 0-100
  status: "compressing" | "uploading" | "done" | "error";
  error?: string;
};
```

Show active uploads below the buttons:
- "compressing" status: show filename with "Compressing..." text and a Loader2 spinning icon
- "uploading" status: show filename with progress bar (h-2 rounded-full bg-zinc-200 with inner div bg-zinc-900 width = progress%)
- "error" status: show filename with error text in red, and a toast via `toast.error()`
- "done" status: remove from visible list (filter out done items)

**Disable both buttons while any upload is in progress** — prevents duplicate taps per research Pitfall 3.

**Error handling:** Wrap each file's pipeline in try/catch. On error, set status to "error", show toast with `toast.error(\`Failed to upload \${filename}\`)`. Continue processing remaining files in the queue.

**Input reset:** After onChange handler processes files, reset the input value to empty string so the same file can be re-selected: `e.target.value = ""`.
